多线程

是不是临界区越小越好，临界区不要涉及太多的函数调用

mutex互斥量是一个类对象，把它理解为一把锁。
std::lock_guard类模板代替lock()和unlock()。

std::async()提供一个接口，能让一段技能或者说一个callable object若是可能的话在后台运行，成为一个独立线程。

Class future<>允许你等待线程结束并获取其结果。

多个线程并发处理相同的数据而又不曾同步化，那么唯一安全的情况就是：所有线程只读取数据。

并发过程中会出现什么样的问题：

  • （未同步化数据访问）：并行运行的两个线程读和写同一笔数据，不知道哪一个语句先来。
  • （写至半途的数据）：某个线程正在读数据，另一个线程改动它，于是读取中的线程甚至可能读到改了一半的数据，读到一个半新半旧的旧值。
  • （重新安排的语句）：语句和操作有可能被重新安排次序，也许对于每一个单线程正确，但对于多个线程的组合却破坏了预期的行为。

**除非另有说明，C++标准库提供的函数通常不支持“写或读”动作与另一个“写”动作（写至同一笔数据）并发执行。

并发问题中，对数据的修改是很敏感的。
重新安排语句的前提是，如果是在单线程下，不影响该线程的行为。
默认情况下，C++编译器应该有能力生成高度优化代码，而某些优化行为需要重新安排语句。

解决问题所需要的性质：
Atomicity（不可切割性）：其行为是独占的，排他的，无任何打断。
Order（次序）：需要一些方法保证”具体指定语句“的次序。

  • 你可以使用condition variable有效地令某线程等待若干”被另一线程控制的“判断式称为true。这有助于应付多线程间的次序，允许一或多个线程处理其他一或多个线程所提供的数据或状态。
  • 可以使用atomic data type确保每次对变量或对象的访问动作都是不可切割的——只要atomic type上的操作次序保证稳定。

递归Lock，借着recursive_mutex，这个mutex允许同一线程多次锁定，并在最近一次（last)相应的unlock()时释放lock。——调用会不会重新占用锁。
尝试性的Lock以及带时间性的Lock
try_lock()==false;
处理多个Lock
全局函数std::lock()会锁住它收到的所有mutex，而且阻塞直到所有mutex都被锁定或者抛出异常，如果是后者，已被锁的会被解锁。使用lock_guard时，并以adopt_lock作为初始化的第二实参，确保任何情况下这些mutex在离开作用域时会被解锁。

以此相同方式你可以尝试”取得多个lock“且”若并非所有lock都可用也不至于造成阻塞“。全局函数std::try_lock()会在取得所有lock情况下返回-1；否则返回第一个失败的lock的索引，且如果这样的话所有成功的lock会又被unlocked。

std::lock()和std::try_lock()要与lock_guard\ lg(m1,std::adopt_lock)一起使用。

还提供class unique_lock<>，它对付mutex更有弹性。它又允许明确写出”何时“以及”如何“锁定或者解锁mutex.

类型 效果
defer_lock 不获得互斥的所有权
try_to_lock 尝试获得互斥的所有权而不阻塞
adopt_lock 假设调用方线程已拥有互斥的所有权

此外，class unique_lock<>提供release()用来释放mutex，或是将其mutex拥有权转移给另一个lock。

对于”某一满足条件“的轮询通常不是好办法。更好的做法是使用condition variable（条件变量）。

C++标准库提供一下mutex Class：

  • Class std::mutex,同一时间只可被一个线程锁定。如果它被锁定，任何其他lock()都会阻塞，直到这个mutex再次可用，且try_lock()会失败。
  • Class std::recursive_mutex，允许在同一时间多次被同一线程获得其lock。其典型应用是：函数捕获一个lock并调用另一函数而后者再次捕获相同的lock。
  • Class std::timed_mutex 额外允许你传递一个时间段或时间点，用来定义多长时间内它可以尝试捕捉一个lock。为此它提供了try_lock_for()和try_lock_until();
  • Class std::recursive_timed_mutex

条件变量

Future允许你停下来，直到另一个线程提供某笔数据或直到另一线程结束。然而future从某线程传递数据到另一线程只能一次。事实上future的主要目的是处理线程的返回值或异常。
条件变量它可用来同步化线程之间的数据流逻辑依赖关系。它是个变量，借由他，一个线程可以唤醒一或多个其他等待中的线程。

Atomic

atomic所提供的操作将使用默认保证，不论内存访问次序如何。这个默认保证提供了顺序一致性，意思是在线程之中atomic操作保证一定“像代码出现的次序”那样地发生。因此重排语句也不会出现。


处理atomic的两个最重要语句是store()和load();

  • store()赋予一个新值。
  • load()取当前值。
